An abstract class is a class that cannot be instantiated (you cannot create its object) and is meant to act as a base class.

👉 It is used when you want to define a common structure but force derived classes to implement specific behavior.

✔ Key Point:

A class becomes abstract if it has at least one pure virtual function.
🔷 What is a Pure Virtual Function?

A pure virtual function is a function that:

Has no implementation in the base class
Must be overridden in derived class
Syntax:
virtual void draw() = 0;

👉 The = 0 means:

“I am not providing implementation”
“Child class MUST implement this”

✅ 1. Cannot create object of abstract class
Shape s; // ❌
✅ 2. Pointer and refrence variables of abstract class is allowed
Shape* s;

✔ Used for runtime polymorphism

✅ 3. Derived class MUST override all pure virtual functions

If not:

class Circle : public Shape {
    // no draw()
};

👉 Then Circle also becomes abstract

✅ 4. Abstract class CAN have:
Normal functions
Constructors
Data members
✅ 5. Pure virtual function CAN have definition (advanced)
void Shape::draw() {
    cout << "Base draw";
}

But still:

virtual void draw() = 0;
    #include <iostream>
    using namespace std;

    class Shape {
    public:
        virtual void draw() = 0;  // pure virtual
    };

    void Shape::draw() {
        cout << "Drawing base shape\n";
    }

    class Circle : public Shape {
    public:
        void draw() {
            cout << "Circle draw start\n";

            Shape::draw();  // calling base version

            cout << "Circle draw end\n";
        }
    };

    int main() {
        Circle c;
        c.draw();
    }
    Output
    Circle draw start
    Drawing base shape
    Circle draw end

🔷 Why do we use Abstract Classes?

👉 To achieve:

Abstraction
Runtime polymorphism
Design enforce rules

Inheritance

⚠️ TRAPS in Inheritance (PLACEMENT GOLD ⚡)

These are exactly what companies ask.

🔴 Trap 1: Private Members Not Accessible
class A {
private:
    int x = 10;
};

class B : public A {
public:
    void show() {
        cout << x; // ❌ ERROR
    }
};

✔ Fix: use protected or getter

🔴 Trap 2: Constructor is NOT Inherited
class A {
public:
    A(int x) {}
};

class B : public A {
};

👉 This will cause error

✔ Fix:

class B : public A {
public:
    B() : A(10) {}
};
🔴 Trap 3: Function Hiding (VERY COMMON)
class A {
public:
    void show() { cout << "A"; }
};

class B : public A {
public:
    void show(int x) { cout << "B"; }
};
B obj;
obj.show(); // ❌ ERROR

👉 Why?
Because B::show(int) hides A::show()

✔ Fix:

using A::show;
🔴 Trap 4: Base Pointer Behavior
class A {
public:
    void show() { cout << "A"; }
};

class B : public A {
public:
    void show() { cout << "B"; }
};

A* ptr = new B();
ptr->show();

👉 Output: A

✔ Why?
Because function is not virtual

🔴 Trap 5: Virtual Function Fix
class A {
public:
    virtual void show() { cout << "A"; }
};

👉 Now output becomes: B

✔ This is runtime polymorphism

🔴 Trap 6: Object Slicing (VERY IMPORTANT ⭐)
class A {
public:
    int x = 10;
};

class B : public A {
public:
    int y = 20;
};

B obj;
A a = obj; // slicing

👉 y is LOST

✔ Only base part copied

🔴 Trap 7: Diamond Problem (Multiple Inheritance)
class A {
public:
    int x;
};

class B : public A {};
class C : public A {};

class D : public B, public C {
};

👉 D has 2 copies of A

✔ Problem:

D obj;
obj.x; // ❌ ambiguous
✔ Fix: Virtual Inheritance
class B : virtual public A {};
class C : virtual public A {};
🔴 Trap 8: Order of Constructor Calls
class A {
public:
    A() { cout << "A"; }
};

class B : public A {
public:
    B() { cout << "B"; }
};
B obj;

👉 Output:

A B

✔ Base class constructor runs FIRST

🔴 Trap 9: Destructor Order

👉 Reverse order:

B destructor
A destructor
🔴 Trap 10: Overriding vs Overloading

👉 Confusion area

Concept	Meaning
Overloading	Same function name, different parameters
Overriding	Redefining base class virtual function
